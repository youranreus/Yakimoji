# Product Requirements Document - Yakimoji

**Author:** 季悠然
**Date:** 2026-05-18

## Executive Summary

Yakimoji 是一个面向高频汉化型个人创作者的频道配方工作台，目标是把视频译制流程中反复出现的默认决策沉淀为可复用资产。它服务的不是低频、一次性的翻译需求，而是持续处理固定来源频道内容、追求稳定产能与更短启动路径的一人内容工厂。产品要解决的核心问题，不是“如何完成一次视频翻译”，而是“如何让同类视频任务在导入后尽快进入可交付状态”，从而减少用户在语言对、字幕模板、输出偏好等环节上的重复配置成本。

Yakimoji 的第一阶段聚焦一条克制而直接的主路径：用户导入 YouTube 链接或上传视频，系统识别来源频道并自动命中已有频道配方，带出默认翻译方向、字幕样式与输出规则，用户仅在必要时进行少量任务级覆盖，然后进入转录、翻译、字幕生成与视频烤制，最终交付成品视频，并附带字幕文件。产品体验应围绕“高命中率、少打扰、快速开工”展开，仅在陌生频道首次出现或低置信度片段需要人工确认时打断用户流程。

### What Makes This Special

Yakimoji 的差异化不在于提供转录、翻译或字幕烤制这些已逐渐商品化的能力，而在于把“来源频道对应的一整套默认工作流”设计为产品核心。对目标用户而言，真正持续消耗时间和心智的并不是单次生成能力本身，而是每次任务启动前重复做出的那些微决策。Yakimoji 通过频道配方把这些决策前置并沉淀下来，让系统在识别到来源后优先替用户完成大部分默认选择。

这种设计让 Yakimoji 更像一个熟练工工作台，而不是一个要求用户每次重新配置参数的通用工具后台。用户感知到的关键差异时刻，不是“这个工具也能翻译视频”，而是“它已经知道我处理这个频道时通常怎么做”。这一点直接决定了任务启动速度、工作节奏连续性，以及用户是否愿意把产品作为长期固定入口。

## Project Classification

Yakimoji 当前被定义为一个 `web_app` 类型的 greenfield 产品，面向创作者工作流与媒体生产场景。其领域复杂度判断为 `medium`：虽然不属于强监管行业，但产品涉及来源识别、任务流编排、频道配方资产、媒体处理链路，以及围绕低打扰体验构建的信息与交互设计，因此需要在工作流效率、可复用性与产品清晰度之间取得平衡。

## Success Criteria

### User Success

目标用户在导入一个已建立频道配方的来源视频后，能够明显感受到系统已替其完成大部分默认决策，任务创建不再需要重复配置语言对、字幕模板与输出偏好。对用户而言，“完成”不只是任务提交成功，而是能在低打扰前提下拿到可直接发布的成品视频，并把字幕文件作为附带资产获取。

用户成功的关键体验信号包括：已命中频道配方的任务能够在 5 分钟内完成导入并进入处理状态；用户仅需执行极少量任务级覆盖；除陌生频道首次出现或低置信度片段确认外，流程不主动打断用户。产品应让用户感到“这次又不用重新配一遍”，而不是“我又要从头设一次参数”。

### Business Success

第一阶段的产品成立标准，不是泛化用户规模，而是能否服务好一小批高频、高价值创作者。短期业务成功的判断标准是：至少获得 10 个重度用户，并且这些用户持续将 Yakimoji 作为固定工作流入口使用，而不是偶发尝试工具。

业务上最关键的北极星指标是配方复用率。若用户为多个来源频道维护配方，并在后续任务中持续复用，说明“频道配方”确实构成了稳定资产。配方复用率应优先于总用户数、注册量或浅层活跃度，作为第一阶段最核心的价值验证指标。

### Technical Success

系统必须稳定支撑“来源识别 -> 命中配方 -> 快速进入处理”这一主路径。技术成功不只是链路可运行，而是能持续兑现产品承诺：已命中配方的任务在 5 分钟内进入处理状态，且该过程具备清晰、可预期、少打扰的行为表现。

技术侧还应保证三项底线：频道配方命中逻辑稳定；人工介入边界严格收敛，只保留陌生频道和低置信度片段两类必要打断；成品视频交付链路优先级高于中间资产管理能力，避免 MVP 被编辑器能力或复杂编排能力拖重。

### Measurable Outcomes

第一阶段应至少跟踪以下可量化结果：已命中频道配方的任务，从导入到进入处理状态的时间控制在 5 分钟内；首批获得并持续留存不少于 10 个重度用户；配方复用率成为核心运营指标，并用于衡量用户是否真正把频道配方视为长期资产。

辅助指标可包括：每个重度用户维护的频道配方数量、已创建配方用户的任务启动频率、成品视频交付完成率，以及任务创建阶段的手动覆盖次数。这些指标用于验证产品是否真正减少了重复决策成本，并提升了高频工作流的连续性。

## Product Scope

### MVP - Minimum Viable Product

MVP 必须覆盖一条完整且可验证的主路径：手动导入 YouTube 链接或上传视频，识别来源频道，自动命中并套用频道配方，允许少量任务级覆盖，然后完成转录、翻译、字幕生成、视频烤制，并交付成品视频及附带字幕文件。

MVP 只保留两个必要打断点：陌生频道首次出现时的轻量配方创建，以及低置信度片段的人工确认。MVP 明确不包含自动订阅、批处理编排、逐句重编辑器，以及其他会削弱“快速开工”主张的复杂能力。

### Growth Features (Post-MVP)

Post-MVP 可以围绕“让频道配方资产更高效地放大”展开，包括自动订阅来源频道、轻量批处理、更多平台导出预设、更丰富但仍受控的任务级覆盖能力，以及更成熟的配方管理与复用机制。

这一阶段的目标不是改变产品重心，而是在已验证配方复用价值之后，放大单个用户的处理频率与规模，让 Yakimoji 从手动工作台逐步演进为轻量生产系统。

### Vision (Future)

未来版本的 Yakimoji 可以从频道配方工作台扩展为更完整的来源驱动型内容本地化系统。届时，配方的抽象可从频道延展到系列、来源源头乃至跨平台分发规则，支撑更连续、更自动化的内容生产链路。

但这一愿景不应反向拉高第一阶段范围。第一阶段的任务仍然是验证一个足够锋利的命题：高频汉化创作者是否愿意围绕频道配方建立并持续复用自己的默认工作流资产。

## User Journeys

### Journey 1: 高频汉化创作者的成功路径

林川是一名持续处理固定 YouTube 来源频道内容的个人创作者。他的工作节奏依赖频率和稳定性，最烦的不是等待模型处理，而是每次都要重新设置语言对、字幕模板和输出方式。过去，他通常要在多个工具之间切换，反复确认同样的参数，任务开始前就已经消耗掉注意力。

某天他把一个熟悉频道的新视频链接粘贴进 Yakimoji。系统识别出来源频道，并立即命中他之前维护好的频道配方，自动带出默认翻译方向、字幕模板与输出偏好。林川扫一眼任务卡，只改了一个字幕模板细节，就直接开始处理。

真正的价值发生在这个瞬间：他意识到自己不需要再“重新开工”，而是在继续一条已经被系统记住的工作流。几分钟内任务进入处理状态，后续输出的是可直接发布的成品视频，字幕文件作为附带资产提供。他的情绪从“又要重复配一遍”转为“终于像在用自己的熟练工工作台”。

这个旅程揭示的能力需求包括：来源频道识别、频道配方命中、任务创建页的默认值自动带出、少量任务级覆盖、清晰的处理状态反馈，以及成品视频优先交付。

### Journey 2: 高频汉化创作者的异常路径

林川又导入了一条视频，但这次来源是一个此前没有处理过的新频道。系统没有命中现有配方，而是明确告诉他这是一个陌生来源，并以轻量方式引导他补充最小必要信息，例如语言对、字幕模板和输出偏好。

他不希望被拉进复杂后台，更不想一次性填写很多字段。Yakimoji 只让他完成一份最小频道配方，然后继续进入处理。稍后，系统在某些低置信度片段上请求人工确认，但只展示必要的片段与上下文，而不是把整条任务变成重编辑流程。

这个异常旅程的关键，不是避免所有异常，而是让异常足够轻、足够可恢复。林川依然觉得系统是在帮他继续开工，而不是把他拖进审批系统。

这个旅程揭示的能力需求包括：未命中配方时的降级路径、轻量频道配方创建、低置信度片段确认机制、不中断整体任务流的恢复设计，以及异常状态下的清晰解释。

### Journey 3: 内部运营/管理者的管理路径

早期的 Yakimoji 团队里，负责产品运行的人可能就是创始人本人。某天她打开后台，不是为了做复杂运营分析，而是想快速知道系统是否真的在兑现“频道配方工作台”的承诺。她最关心的是：哪些任务成功命中了配方，哪些频道反复没有命中，哪些任务经常需要人工补充。

她查看一组核心面板：配方复用率、各频道的命中情况、任务从导入到进入处理的耗时，以及失败或打断最多的节点。她发现某几个频道的命中率很低，于是进一步检查是来源识别不稳定，还是配方创建流程过重导致用户没有完成沉淀。

这个角色不需要一个过重的 BI 系统，但需要足够强的运营可见性，来验证 MVP 的核心假设是否成立。她的“aha moment”不是某张漂亮图表，而是终于能明确判断：Yakimoji 带来的价值到底是因为配方复用真的成立，还是只是用户在忍受一个更换了外壳的传统任务工具。

这个旅程揭示的能力需求包括：配方复用率指标、任务漏斗与耗时可视化、命中/未命中原因追踪、频道维度的运营视图，以及最小可用的后台管理能力。

### Journey 4: 支持/排障角色的问题定位路径

一位重度用户发来反馈：“为什么我明明以前跑过这个频道，这次却没命中配方？” 负责支持的人需要在不打扰开发的前提下，先快速定位问题。她打开任务详情，看到这次任务的来源识别结果、是否尝试匹配已有频道、匹配失败发生在哪一层，以及用户当时手动覆盖了哪些字段。

如果是来源识别偏差，她能直接判断这是识别问题；如果是用户曾创建过相似但不一致的频道配方，她能看见冲突点；如果是任务在低置信度确认后被中断，她也能复盘停在哪一步。整个支持过程的目标不是“人工替系统补流程”，而是快速解释失败原因，并帮助用户恢复信任。

这个旅程中的关键情绪是避免无助感。支持角色不能面对一个黑盒，也不能只看到“任务失败”四个字。系统需要给出足够可操作的上下文，才能让问题被解释、被追踪、被修复。

这个旅程揭示的能力需求包括：任务审计信息、来源识别与配方匹配日志、人工覆盖记录、失败原因分类、可供支持查看的任务时间线，以及基本的问题恢复建议。

### Journey 5: API 集成方的自动触发路径

一家小型内容工作室内部有自己的选题和发布系统。他们不想让运营同学每次手动粘贴链接，而是希望在内部系统确认选题后，自动把视频任务推送到 Yakimoji。负责集成的工程师最在意的不是炫酷能力，而是接口是否足够清晰、任务状态是否可查询、失败时是否有明确反馈。

他调用 API 创建任务，传入视频来源信息和必要元数据。Yakimoji 在后台尝试识别来源频道并命中配方；如果命中，任务直接进入处理；如果没有命中，则返回明确状态，让工作室决定是否回到人工补充流程。后续工程师还能查询任务进度、最终结果和失败原因，把 Yakimoji 作为他们生产系统中的一个稳定节点，而不是一个只能手工使用的孤岛工具。

这个旅程的价值在于，它验证 Yakimoji 不只是一个界面友好的工作台，也有潜力成为更大内容生产链路中的能力节点。但在 MVP 阶段，这条旅程应保持克制，只覆盖最核心的任务创建、状态查询与结果获取，不扩张成复杂平台集成方案。

这个旅程揭示的能力需求包括：任务创建 API、任务状态查询、结果获取、未命中配方时的状态返回、失败原因结构化表达，以及与手动流程兼容的降级机制。

### Journey Requirements Summary

这些旅程共同指向几类核心能力。第一类是任务入口能力，包括手动导入、上传、以及 API 创建任务。第二类是配方系统能力，包括来源识别、频道配方命中、轻量配方创建、以及任务级覆盖。第三类是处理与交付能力，包括清晰的任务状态流转、低置信度片段确认、成品视频优先交付、以及字幕文件附带输出。

除此之外，Yakimoji 还需要一组支撑长期可运营的系统能力：后台对配方复用率和命中情况的可视化、支持侧可见的任务审计与失败原因、以及 API 场景下的结构化状态返回。换句话说，用户旅程不仅定义了前台体验，也直接定义了系统必须具备的可观察性、可恢复性与可集成性。
