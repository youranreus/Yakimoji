# Product Requirements Document - Yakimoji

**Author:** 季悠然
**Date:** 2026-05-18

## Executive Summary

Yakimoji 是一个面向高频汉化型个人创作者的频道预设工作台，目标是把视频译制流程中反复出现的默认决策沉淀为可复用资产。它服务的不是低频、一次性的翻译需求，而是持续处理固定来源频道内容、追求稳定产能与更短启动路径的一人内容工厂。产品要解决的核心问题，不是“如何完成一次视频翻译”，而是“如何让同类视频任务在导入后尽快进入可交付状态”，从而减少用户在语言对、字幕模板、输出偏好等环节上的重复配置成本。

Yakimoji 的第一阶段聚焦一条克制而直接的主路径：用户导入 YouTube 链接或上传视频，系统识别来源频道并自动命中已有频道预设，带出默认翻译方向、字幕样式与输出规则，用户仅在必要时进行少量任务级覆盖，然后进入转录、翻译、字幕生成与视频烤制，最终交付成品视频，并附带字幕文件。产品体验应围绕“高命中率、少打扰、快速开工”展开，仅在陌生频道首次出现或低置信度片段需要人工确认时打断用户流程。

### What Makes This Special

Yakimoji 的差异化不在于提供转录、翻译或字幕烤制这些已逐渐商品化的能力，而在于把“来源频道对应的一整套默认工作流”设计为产品核心。对目标用户而言，真正持续消耗时间和心智的并不是单次生成能力本身，而是每次任务启动前重复做出的那些微决策。Yakimoji 通过频道预设把这些决策前置并沉淀下来，让系统在识别到来源后优先替用户完成大部分默认选择。

这种设计让 Yakimoji 更像一个熟练工工作台，而不是一个要求用户每次重新配置参数的通用工具后台。用户感知到的关键差异时刻，不是“这个工具也能翻译视频”，而是“它已经知道我处理这个频道时通常怎么做”。这一点直接决定了任务启动速度、工作节奏连续性，以及用户是否愿意把产品作为长期固定入口。

### Experience Principles

Yakimoji 的首屏体验应围绕“识别来源频道并带出默认任务预设”展开，形成看一眼即可开跑的工作台感。第一阶段不应把首页做成复杂任务创建器或参数后台，而应让用户在进入任务之前就感知到系统已经替其完成大部分默认决策。

字幕模板在产品前台的心智应围绕平台适配展开，而不是让用户直接面对画面安全区、视觉遮挡或复杂版式判断等实现层概念。对用户而言，模板首先是面向发布平台和观看场景的输出选择；更底层的画面安全策略可以存在于系统内部，但不应成为第一阶段用户理解产品的主要方式。

## Project Classification

Yakimoji 当前被定义为一个 `web_app` 类型的 greenfield 产品，面向创作者工作流与媒体生产场景。其领域复杂度判断为 `medium`：虽然不属于强监管行业，但产品涉及来源识别、任务流编排、频道预设资产、媒体处理链路，以及围绕低打扰体验构建的信息与交互设计，因此需要在工作流效率、可复用性与产品清晰度之间取得平衡。

## Success Criteria

### User Success

目标用户在导入一个已建立频道预设的来源视频后，能够明显感受到系统已替其完成大部分默认决策，任务创建不再需要重复配置语言对、字幕模板与输出偏好。对用户而言，“完成”不只是任务提交成功，而是能在低打扰前提下拿到可直接发布的成品视频，并把字幕文件作为附带资产获取。

用户成功的关键体验信号包括：已命中频道预设的任务能够在 5 分钟内完成导入并进入处理状态；用户仅需执行极少量任务级覆盖；除陌生频道首次出现或低置信度片段确认外，流程不主动打断用户。产品应让用户感到“这次又不用重新配一遍”，而不是“我又要从头设一次参数”。

### Business Success

第一阶段的产品成立标准，不是泛化用户规模，而是能否服务好一小批高频、高价值创作者。短期业务成功的判断标准是：至少获得 10 个重度用户，并且这些用户持续将 Yakimoji 作为固定工作流入口使用，而不是偶发尝试工具。

业务上最关键的北极星指标是预设复用率。若用户为多个来源频道维护预设，并在后续任务中持续复用，说明“频道预设”确实构成了稳定资产。预设复用率应优先于总用户数、注册量或浅层活跃度，作为第一阶段最核心的价值验证指标。

### Technical Success

系统必须稳定支撑“来源识别 -> 命中预设 -> 快速进入处理”这一主路径。技术成功不只是链路可运行，而是能持续兑现产品承诺：已命中预设的任务在 5 分钟内进入处理状态，且该过程具备清晰、可预期、少打扰的行为表现。

技术侧还应保证三项底线：频道预设命中逻辑稳定；人工介入边界严格收敛，只保留陌生频道和低置信度片段两类必要打断；成品视频交付链路优先级高于中间资产管理能力，避免 MVP 被编辑器能力或复杂编排能力拖重。

### Measurable Outcomes

第一阶段应至少跟踪以下可量化结果：已命中频道预设的任务，从导入到进入处理状态的时间控制在 5 分钟内；首批获得并持续留存不少于 10 个重度用户；预设复用率成为核心运营指标，并用于衡量用户是否真正把频道预设视为长期资产。

辅助指标可包括：每个重度用户维护的频道预设数量、已创建预设用户的任务启动频率、成品视频交付完成率，以及任务创建阶段的手动覆盖次数。这些指标用于验证产品是否真正减少了重复决策成本，并提升了高频工作流的连续性。

## Product Scope

### Current Release Focus

当前发布必须覆盖一条完整且可验证的主路径：手动导入 YouTube 链接或上传视频，识别来源频道，自动命中并套用频道预设，允许少量任务级覆盖，然后完成转录、翻译、字幕生成、视频烤制，并交付成品视频及附带字幕文件。

当前发布只保留两个必要打断点：陌生频道首次出现时的轻量预设创建，以及低置信度片段的人工确认。当前发布明确不包含自动订阅、批处理编排、逐句重编辑器，以及其他会削弱“快速开工”主张的复杂能力。

同时，字幕安全区预警、智能画面遮挡检测等能力应后置处理，不作为第一阶段主动打断用户或影响任务启动的判断机制。对于模板不合适的问题，第一阶段应优先通过任务级字幕模板覆盖解决，而不是引入更重的自动视觉判断。

### Adjacent Expansion Directions

后续扩展可以围绕“让频道预设资产更高效地放大”展开，包括自动订阅来源频道、轻量批处理、更多平台导出预设、更丰富但仍受控的任务级覆盖能力，以及更成熟的预设管理与复用机制。

这一阶段的目标不是改变产品重心，而是在已验证预设复用价值之后，放大单个用户的处理频率与规模，让 Yakimoji 从手动工作台逐步演进为轻量生产系统。

### Long-Term Vision

未来版本的 Yakimoji 可以从频道预设工作台扩展为更完整的来源驱动型内容本地化系统。届时，预设的抽象可从频道延展到系列、来源源头乃至跨平台分发规则，支撑更连续、更自动化的内容生产链路。

但这一愿景不应反向拉高第一阶段范围。第一阶段的任务仍然是验证一个足够锋利的命题：高频汉化创作者是否愿意围绕频道预设建立并持续复用自己的默认工作流资产。

## User Journeys

### Journey 1: 高频汉化创作者的成功路径

林川是一名持续处理固定 YouTube 来源频道内容的个人创作者。他的工作节奏依赖频率和稳定性，最烦的不是等待模型处理，而是每次都要重新设置语言对、字幕模板和输出方式。过去，他通常要在多个工具之间切换，反复确认同样的参数，任务开始前就已经消耗掉注意力。

某天他把一个熟悉频道的新视频链接粘贴进 Yakimoji。系统识别出来源频道，并立即命中他之前维护好的频道预设，自动带出默认翻译方向、字幕模板与输出偏好。林川扫一眼任务卡，只改了一个字幕模板细节，就直接开始处理。

真正的价值发生在这个瞬间：他意识到自己不需要再“重新开工”，而是在继续一条已经被系统记住的工作流。几分钟内任务进入处理状态，后续输出的是可直接发布的成品视频，字幕文件作为附带资产提供。他的情绪从“又要重复配一遍”转为“终于像在用自己的熟练工工作台”。

这个旅程揭示的能力需求包括：来源频道识别、频道预设命中、任务创建页的默认值自动带出、少量任务级覆盖、清晰的处理状态反馈，以及成品视频优先交付。

### Journey 2: 高频汉化创作者的异常路径

林川又导入了一条视频，但这次来源是一个此前没有处理过的新频道。系统没有命中现有预设，而是明确告诉他这是一个陌生来源，并以轻量方式引导他补充最小必要信息，例如语言对、字幕模板和输出偏好。

他不希望被拉进复杂后台，更不想一次性填写很多字段。Yakimoji 只让他完成一份最小频道预设，然后继续进入处理。稍后，系统在某些低置信度片段上请求人工确认，但只展示必要的片段与上下文，而不是把整条任务变成重编辑流程。

这个异常旅程的关键，不是避免所有异常，而是让异常足够轻、足够可恢复。林川依然觉得系统是在帮他继续开工，而不是把他拖进审批系统。

这个旅程揭示的能力需求包括：未命中预设时的降级路径、轻量频道预设创建、低置信度片段确认机制、不中断整体任务流的恢复设计，以及异常状态下的清晰解释。

### Journey 3: 内部运营/管理者的管理路径

早期的 Yakimoji 团队里，负责产品运行的人可能就是创始人本人。某天她打开后台，不是为了做复杂运营分析，而是想快速知道系统是否真的在兑现“频道预设工作台”的承诺。她最关心的是：哪些任务成功命中了预设，哪些频道反复没有命中，哪些任务经常需要人工补充。

她查看一组核心面板：预设复用率、各频道的命中情况、任务从导入到进入处理的耗时，以及失败或打断最多的节点。她发现某几个频道的命中率很低，于是进一步检查是来源识别不稳定，还是预设创建流程过重导致用户没有完成沉淀。

这个角色不需要一个过重的 BI 系统，但需要足够强的运营可见性，来验证 MVP 的核心假设是否成立。她的“aha moment”不是某张漂亮图表，而是终于能明确判断：Yakimoji 带来的价值到底是因为预设复用真的成立，还是只是用户在忍受一个更换了外壳的传统任务工具。

这个旅程揭示的能力需求包括：预设复用率指标、任务漏斗与耗时可视化、命中/未命中原因追踪、频道维度的运营视图，以及最小可用的后台管理能力。

### Journey 4: 支持/排障角色的问题定位路径

一位重度用户发来反馈：“为什么我明明以前跑过这个频道，这次却没命中预设？” 负责支持的人需要在不打扰开发的前提下，先快速定位问题。她打开任务详情，看到这次任务的来源识别结果、是否尝试匹配已有频道、匹配失败发生在哪一层，以及用户当时手动覆盖了哪些字段。

如果是来源识别偏差，她能直接判断这是识别问题；如果是用户曾创建过相似但不一致的频道预设，她能看见冲突点；如果是任务在低置信度确认后被中断，她也能复盘停在哪一步。整个支持过程的目标不是“人工替系统补流程”，而是快速解释失败原因，并帮助用户恢复信任。

这个旅程中的关键情绪是避免无助感。支持角色不能面对一个黑盒，也不能只看到“任务失败”四个字。系统需要给出足够可操作的上下文，才能让问题被解释、被追踪、被修复。

这个旅程揭示的能力需求包括：任务审计信息、来源识别与预设匹配日志、人工覆盖记录、失败原因分类、可供支持查看的任务时间线，以及基本的问题恢复建议。

### Journey 5: API 集成方的自动触发路径

一家小型内容工作室内部有自己的选题和发布系统。他们不想让运营同学每次手动粘贴链接，而是希望在内部系统确认选题后，自动把视频任务推送到 Yakimoji。负责集成的工程师最在意的不是炫酷能力，而是接口是否足够清晰、任务状态是否可查询、失败时是否有明确反馈。

他调用 API 创建任务，传入视频来源信息和必要元数据。Yakimoji 在后台尝试识别来源频道并命中预设；如果命中，任务直接进入处理；如果没有命中，则返回明确状态，让工作室决定是否回到人工补充流程。后续工程师还能查询任务进度、最终结果和失败原因，把 Yakimoji 作为他们生产系统中的一个稳定节点，而不是一个只能手工使用的孤岛工具。

这个旅程的价值在于，它验证 Yakimoji 不只是一个界面友好的工作台，也有潜力成为更大内容生产链路中的能力节点。但在 MVP 阶段，这条旅程应保持克制，只覆盖最核心的任务创建、状态查询与结果获取，不扩张成复杂平台集成方案。

这个旅程揭示的能力需求包括：任务创建 API、任务状态查询、结果获取、未命中预设时的状态返回、失败原因结构化表达，以及与手动流程兼容的降级机制。

### Journey Requirements Summary

这些旅程共同指向几类核心能力。第一类是任务入口能力，包括手动导入、上传、以及 API 创建任务。第二类是预设系统能力，包括来源识别、频道预设命中、轻量预设创建、以及任务级覆盖。第三类是处理与交付能力，包括清晰的任务状态流转、低置信度片段确认、成品视频优先交付、以及字幕文件附带输出。

除此之外，Yakimoji 还需要一组支撑长期可运营的系统能力：后台对预设复用率和命中情况的可视化、支持侧可见的任务审计与失败原因、以及 API 场景下的结构化状态返回。换句话说，用户旅程不仅定义了前台体验，也直接定义了系统必须具备的可观察性、可恢复性与可集成性。

## Web 应用特定要求

### 项目类型概述

Yakimoji 第一阶段应被实现为一个以登录后工作台为核心的单页 Web 应用（SPA）。产品的主要使用场景是高频任务处理、频道预设复用、任务状态跟踪与异常确认，因此核心体验应围绕持续操作效率而非页面跳转式浏览展开。主生产环境以现代桌面浏览器为主，移动端不是主要创作端，但应支持关键查看与轻操作场景，以便用户在非桌面环境下仍可掌握任务进度并完成必要处理。

该产品不是内容发布站点，也不以搜索获取流量为第一阶段目标，因此产品本体不应为 SEO 做额外结构性妥协。Web 形态的价值主要体现在跨设备可访问、处理链路可视化、工作台式任务管理，以及围绕频道预设形成持续复用的任务入口。

### 技术架构考量

前端架构应以 SPA 为前提，优先保证登录后工作台、任务列表、任务详情、频道预设管理与异常确认流程的一致性与流畅性。由于 Yakimoji 的主要用户是高频创作者，系统的核心技术目标不是炫技式实时交互，而是让任务状态变化、处理进度与待确认节点能被稳定、低打扰地同步给用户。

实时能力建议采用 SSE 作为第一优先方案，用于推送任务状态更新、进度变化、失败事件与低置信度确认提醒。这一方案较适合以服务端单向事件通知为主的任务型工作流，也有利于控制实现复杂度。若某些部署环境或链路条件不适合 SSE，则应提供轮询作为兜底机制，确保状态同步不会因为实时通道问题而失效。

系统还应明确桌面优先的交互策略。任务创建、任务详情浏览、成品下载、低置信度片段确认等核心流程应在桌面端获得完整支持；移动端则应至少支持查看任务列表、查看任务详情、下载成品，以及处理低置信度确认，从而满足用户在外部场景下的轻量跟进需求。

### 浏览器支持矩阵

第一阶段应以现代桌面浏览器为正式支持范围，至少覆盖当前主流版本的 Chrome、Edge、Safari 等浏览器。Firefox 可视为兼容性目标，但不必为了少数非关键差异显著增加实现复杂度。浏览器支持策略应优先围绕目标用户的实际使用环境，而不是追求过宽覆盖。

移动端浏览器支持应以“可查看、可确认、可下载”为标准，而不是完整复刻桌面生产体验。也就是说，移动端需要保证任务列表、任务详情、低置信度确认与成品下载的基本可用性，但不要求承担完整的高频创作工作流。

### 响应式设计要求

响应式策略应体现明显的桌面优先思路。桌面端负责承载完整工作台体验，包括任务表格或卡片列表、详情侧栏或详情页、频道预设入口、状态筛选与下载操作。移动端应采用压缩后的信息结构，优先保证关键信息与关键动作的可达性，而不是保留所有桌面交互密度。

在设计上，应避免把移动端做成一个功能残缺却难以使用的迷你后台。更合理的方式是明确其职责边界：查看状态、进入详情、处理必要确认、执行下载。与复杂参数配置、批量操作、长时间停留编辑相关的能力，应继续以桌面端为主。

### 性能目标

Yakimoji 作为高频工作台，其性能要求应直接服务于“快速开工”和“持续顺手”这两个核心体验目标。登录后进入工作台的首屏应尽量在 2 秒内达到可交互状态，至少能够完成任务列表查看、基础筛选与进入详情等核心动作。

日常操作不应出现明显卡顿，尤其是在任务列表切换、详情打开、状态刷新、下载触发与确认操作等高频路径上。长列表场景必须支持分页，避免一次性加载过多任务导致前端性能下降。随着任务量上升，前端结构应能够平稳承接更多历史任务与处理记录，而不让“工作台”逐步退化成迟缓后台。

### SEO 策略

第一阶段产品本体不需要以 SEO 为目标。Yakimoji 的核心价值发生在登录后工作流内部，而不是公开页面的搜索发现，因此任务页、详情页与预设管理页不需要为搜索引擎索引进行专门设计。

如果未来存在品牌官网、营销页或产品介绍页，这些内容可以作为独立资产单独考虑 SEO，但不应影响主产品架构与前端实现判断。换句话说，SEO 不应成为第一阶段 Web 产品体验设计的约束条件。

### 无障碍级别

第一阶段应以 WCAG AA 为目标级别，至少覆盖对高频工作流影响最大的无障碍基础能力。这包括：清晰可见的键盘焦点、合理的文本与背景对比度、表单与按钮的可识别标签、状态变化的可感知反馈，以及在错误、处理中、待确认等关键状态下的明确提示。

无障碍在 Yakimoji 中不仅是合规问题，也与效率直接相关。对于一个高频使用的工作台产品，如果导航、状态反馈和关键动作的可感知性做得差，就会直接拖慢熟练用户的操作节奏。因此，应把无障碍视为交互质量基线，而不是发布后补做的装饰项。

### 实施考量

实现层面应坚持“桌面优先、移动可跟进、实时轻量、性能稳定”的四个原则。前端实现不需要为第一阶段引入过重的复杂机制，但应在架构上预留任务状态推送、分页扩展、详情视图深化和后续多端能力增强的空间。

同时，Web 应用的具体实现应始终服务于 Yakimoji 的产品差异，而不是成为抽象技术选型展示。也就是说，SPA、SSE、响应式与无障碍要求，最终都应回到同一个产品目标：让用户在识别来源、命中预设、查看进度、处理异常和下载成品的全过程中，感受到一个顺手、稳定、低打扰的工作台体验。

## Project Scoping

### Strategy & Philosophy

**Approach:** 这次发布采用单次发布策略，而非分阶段交付。范围定义的核心原则不是尽量多做功能，而是在一次发布中把 Yakimoji 的主价值链条完整跑通，并确保该链条在手动入口与外部 API 入口下都可成立。产品判断的重点不是“是否覆盖所有视频本地化能力”，而是“频道预设工作台”这一主路径能否作为一个稳定、顺手、可反复使用的工作流入口真正成立。

本次范围控制应坚持三条原则。第一，核心链路不可拆断，即从导入或上传开始，到来源识别、频道预设命中或创建、少量任务级覆盖、处理状态跟踪、低置信度确认，再到成品交付，必须形成闭环。第二，外部 API 不是附属能力，而是正式范围的一部分，必须支持任务创建、状态查询与结果获取。第三，运营与支持能力只做到最小可用，不把本次发布拉向重型后台或复杂运营平台。

**Resource Requirements:** 按当前判断，最小可上线团队可以压缩为一名具备前后端实现能力、并能承担基础产品与设计判断的通才型成员。这个资源假设极度克制，意味着范围必须高度收敛，且所有非核心增强项都要让位于主链路稳定性、状态可见性与交付可靠性。如果后续资源条件改善，应优先补足测试、设计细化与系统稳定性保障，而不是继续横向扩展功能。

### Complete Feature Set

**Core User Journeys Supported:** 本次单次发布必须完整支撑三类关键旅程。第一类是高频汉化创作者的主路径：用户导入 YouTube 链接或上传视频，系统识别来源频道，命中已有频道预设或创建最小预设，允许通过字幕模板进行少量任务级覆盖，随后进入处理流程，用户可查看状态、在必要时处理低置信度片段，并最终下载成品视频与附带字幕文件。第二类是支持与排障的最小路径：系统应提供最基本的任务审计、命中结果、失败原因与时间线可见性，使支持者能够解释“为什么没有命中预设”或“为什么任务没有顺利完成”。第三类是 API 集成路径：外部系统应能够创建任务、查询任务状态并获取结果，在未命中预设或任务失败时也能收到结构化反馈。

**Must-Have Capabilities:** 本次发布的必须能力包括：手动导入 YouTube 链接与上传视频；来源频道识别；频道预设的最小创建、命中与复用；任务级覆盖仅支持字幕模板；转录、翻译、字幕生成、视频烤制与成品视频交付的完整处理链路；字幕文件作为附带资产输出；任务状态的稳定展示与轻量实时同步；低置信度片段确认流程；桌面端完整工作台能力；移动端对任务详情、成品下载与低置信度确认的基本支持；外部 API 的任务创建、状态查询与结果获取；未命中预设与失败原因的结构化返回；以及最小可用的运营/支持视图，包括任务审计、命中情况与基础时间线信息。

**Nice-to-Have Capabilities:** 在不影响主链路闭环的前提下，本次单次发布中的次优先能力可包括：更细的浏览器兼容性打磨，例如对 Firefox 的更完整适配；更丰富的运营可视化，例如频道维度汇总面板与更细的漏斗指标；更完善的支持侧解释信息与恢复建议；更流畅的移动端交互细节；以及在不增加复杂度的前提下，对任务列表筛选、历史记录浏览和状态提醒体验的进一步优化。这些能力可以进入本次发布范围，但其优先级必须低于主链路稳定性与 API 可用性。

### Risk Mitigation Strategy

**Technical Risks:** 当前最大的技术风险在于处理链路编排复杂、状态同步容易混乱，以及视频处理与成品交付链路的可靠性不足。为降低这类风险，第一，系统必须把任务生命周期建模为清晰、有限、可审计的状态流转，而不是依赖隐式流程拼接。第二，实时状态同步应优先采用 SSE，并以轮询兜底，确保前端状态展示不会因推送机制问题而失真。第三，成品视频交付链路应被视为比中间资产管理更高优先级的稳定性目标，任何会影响交付可靠性的增强功能都不应优先于主链路修稳。

**Market Risks:** 当前这一轮范围界定不把市场风险作为主要裁剪依据。也就是说，本次发布优先围绕产品路径成立与链路可用性展开，而不是围绕市场试探性最小化。市场验证仍然重要，但不会作为本轮单次发布删减核心能力的主要理由。

**Resource Risks:** 在资源极度紧张、最小团队仅能维持单人通才交付的前提下，最大的资源风险不是功能数量不足，而是上下游链路都需要同一人承担，导致系统稳定性、测试深度与交付节奏相互挤压。应对方式是进一步收敛任务级覆盖字段，只保留字幕模板；把运营与支持能力严格控制在最小可用；避免引入复杂自动化订阅、重型后台和逐句编辑器等会显著拖重交付的能力。若资源进一步收缩，优先保住核心链路闭环与 API 可用性，而不是保留更多边缘增强项。

## Functional Requirements

### Task Intake & Task Creation

- FR1: 创作者可以通过提交 YouTube 链接创建视频处理任务。
- FR2: 创作者可以通过上传视频文件创建视频处理任务。
- FR3: 创作者可以在任务创建过程中查看系统识别到的来源信息。
- FR4: 创作者可以在命中频道预设时，以已带出默认配置的方式创建任务。
- FR5: 创作者可以在未命中频道预设时，通过最小补充流程继续创建当前任务。
- FR6: 外部系统可以通过 API 创建视频处理任务。
- FR7: 创作者可以查看任务创建前将要生效的关键任务设置。
- FR8: 创作者可以访问自己的任务工作台并查看已创建任务。

### Channel Preset Management

- FR9: 创作者可以为新的来源频道创建频道预设。
- FR10: 创作者可以在频道预设中定义默认翻译方向。
- FR11: 创作者可以在频道预设中定义默认字幕模板。
- FR12: 创作者可以在频道预设中定义默认输出偏好。
- FR13: 创作者可以编辑已有频道预设。
- FR14: 创作者可以查看自己已维护的频道预设。
- FR15: 系统可以在来源匹配成功时自动复用对应频道预设。
- FR16: 系统可以明确标识任务是否命中了已有频道预设、创建了新预设，或未使用预设。

### Task Configuration & Processing

- FR17: 创作者可以为单个任务覆盖默认字幕模板。
- FR18: 创作者可以提交任务进入完整的视频处理流程。
- FR19: 系统可以对任务执行转录、翻译、字幕生成、视频烤制与结果产出。
- FR20: 创作者可以查看任务当前所处的处理状态。
- FR21: 创作者可以查看任务从开始到完成或失败的状态流转过程。
- FR22: 创作者可以查看任务详情，包括当前状态、来源信息与处理结果概览。
- FR23: 系统可以在任务失败或中断时向创作者提供明确的任务结果状态。

### Review & Exception Handling

- FR24: 系统可以在来源频道未命中现有预设时，提示创作者执行以下处理之一：创建新的最小频道预设、为当前任务选择一个已有频道预设，或在不保存频道预设的前提下继续当前任务。
- FR25: 创作者可以在不中断当前任务目标的前提下创建最小频道预设。
- FR26: 系统可以识别需要人工处理的低置信度片段。
- FR27: 创作者可以查看需要人工确认的低置信度片段及其相关上下文。
- FR28: 创作者可以对低置信度片段进行确认或处理，并让任务继续推进。
- FR29: 创作者可以仅在以下异常场景下被要求人工介入任务：来源频道未命中现有频道预设，或系统识别出需要人工确认的低置信度片段。
- FR30: 支持人员可以查看任务未命中频道预设的原因。
- FR31: 支持人员可以查看任务处理失败或中断的原因分类。
- FR32: 支持人员可以查看任务处理过程中的关键时间线与上下文信息。
- FR33: 支持人员可以查看任务曾使用的人工覆盖与人工确认记录。

### Delivery & Result Access

- FR34: 创作者可以获取已完成任务的成品视频。
- FR35: 创作者可以获取与成品任务关联的字幕文件。
- FR36: 创作者可以查看单个任务的最终交付结果状态。
- FR37: 创作者可以在移动端浏览器中查看任务列表与任务详情。
- FR38: 创作者可以在移动端浏览器中处理低置信度确认。
- FR39: 创作者可以在移动端浏览器中下载已完成任务的交付物。

### External API Capabilities

- FR40: 外部系统可以查询任务当前状态。
- FR41: 外部系统可以获取任务完成后的结果信息。
- FR42: 外部系统可以获取任务失败、中断或未命中预设时的结构化结果。
- FR43: 外部系统可以区分任务是否已进入处理、等待人工处理、处理失败或处理完成。
- FR44: 外部系统可以把 Yakimoji 作为可查询状态和结果的处理节点接入自身工作流。

### Operations & Minimal Administrative Visibility

- FR45: 运营或管理角色可以查看任务是否成功命中频道预设。
- FR46: 运营或管理角色可以查看哪些来源频道反复未命中预设。
- FR47: 运营或管理角色可以查看任务从导入到进入处理以及最终完成的关键耗时信息。
- FR48: 运营或管理角色可以查看任务在哪些环节发生失败、中断或人工介入。
- FR49: 运营或管理角色可以查看频道预设复用情况。
- FR50: 系统可以为任务提供最小审计记录，至少包含任务 ID、来源标识、命中的频道预设或未命中结果、任务状态流转时间戳、人工覆盖记录、人工确认记录，以及失败或中断原因，供运营、支持与排障使用。

## Non-Functional Requirements

### Performance

- NFR1: 登录后工作台首页在正常网络条件下（下行带宽不少于 10 Mbps，网络 RTT 不高于 100 ms）应在 95 百分位于 2 秒内达到可交互状态，并以前端真实用户监测或等效性能监测验证。
- NFR2: 在当前页面所需数据已返回的前提下，任务列表浏览、筛选、进入详情与触发下载等高频用户操作应在 95 百分位于 300 毫秒内完成可感知响应，并以前端交互时延监测验证。
- NFR3: 长列表场景必须支持分页，单页最多展示 50 个任务；在不少于 10,000 条历史任务的数据规模下，任务列表首屏加载仍须满足 NFR1，且分页切换仍须满足 NFR2。

### Reliability

- NFR4: 当任务进入失败终态后，系统必须在 30 秒内向创作者界面和外部 API 暴露失败状态，并返回机器可读的失败原因代码与可读失败说明。
- NFR5: 当成品视频或相关交付物生成完成后，系统必须在至少 7 天内保持可下载，并在滚动 30 天内维持不低于 99% 的下载成功率，以下载日志与结果访问监测验证。
- NFR6: 系统对外展示的任务状态在 95% 的状态流转中应于后台真实状态变化后 30 秒内完成同步，且任何单次状态不同步不得超过 60 秒，并以后台状态日志与前台或 API 状态对账验证。
- NFR7: 当任务发生失败、中断或人工介入时，系统必须保留至少 30 天的状态与审计信息；这些记录至少包含 FR50 定义的字段，并支持用户、支持人员和外部系统按任务 ID 查询。

### Security

- NFR8: 所有未认证请求访问受保护页面或资源时，必须被重定向到登录流程或返回 401/403；在自动化访问测试中，受保护资源被未认证访问成功的比例必须为 0。
- NFR9: 所有外部 API 在凭证缺失、无效或过期时，必须在 100% 的认证合约测试用例中返回 401 或 403，且不得返回受保护业务数据。
- NFR10: 在以“任务创建者”“同组织支持人员”“无权限用户”三类角色执行的授权测试中，用户对非本人任务、结果文件与相关记录的越权访问成功率必须为 0；所有越权请求必须返回 403 或 404，并记录请求主体、目标任务 ID 与拒绝时间戳。
- NFR11: 任务相关文件与结果资产不得通过匿名长期公共 URL 直接访问；所有交付物访问链接必须具备身份校验或时效控制，默认有效期不得超过 24 小时；在未授权访问测试中，未授权请求成功率必须为 0，并保留访问失败日志。
- NFR12: 第一阶段发布前，发布检查单必须逐项确认 NFR8-NFR11 对应控制项已通过自动化或人工验收，并明确记录“正式外部合规认证不在本阶段发布门槛内”；若上述记录缺失，则本阶段版本不得发布。

### Integration

- NFR13: 对外 API 的任务状态语义必须在同一 API 版本内保持向后兼容；任何状态枚举或字段语义变更都必须通过新版本或显式弃用流程发布，并通过 100% 的接口合约测试。
- NFR14: 对外 API 在失败、中断或未命中预设等异常场景下，必须返回统一的错误响应结构，至少包含任务 ID、状态、原因代码与可读说明，并通过 100% 的异常场景合约测试。
- NFR15: 所有任务结果获取接口必须返回统一的结果响应结构，至少包含任务 ID、终态状态、结果元数据与交付物访问方式；不同任务结果类型不得使用彼此不兼容的响应 envelope，并通过 100% 的结果合约测试。
