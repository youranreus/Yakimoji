## Executive Summary

Epic 2 is complete from the sprint tracker perspective. All four Epic 2 stories are marked `done` in `sprint-status.yaml`:

- `2-1-create-and-manage-channel-presets`
- `2-2-resolve-preset-match-for-familiar-sources`
- `2-3-unknown-source-manual-resolution`
- `2-4-task-level-subtitle-template-override-on-top-of-presets`

The epic delivered the full preset-backed task launch foundation: creator-owned channel presets, automatic familiar-source matching, explicit unknown-source manual resolution, and task-level subtitle template overrides on top of preset defaults.

The main retrospective finding is not about missing Epic 2 functionality. The main finding is that Epic 3 must be started with stronger cross-story contracts for human intervention, failure explanation, retry attempts, and support diagnostics. Epic 2 proved that context retention and semantic provenance are fragile areas; Epic 3 depends on those areas even more heavily.

## Epic 2 Outcomes

### What Went Well

- The preset domain landed as a durable server-side resource, not front-end temporary state. Story 2.1 added owner-scoped persistence, uniqueness by creator/source, service contracts, workspace UI, and regression coverage.
- Familiar-source matching was implemented with stable snapshot semantics. Story 2.2 preserved task creation-time preset state instead of letting later preset edits silently mutate historical tasks.
- Unknown-source handling became a formal product branch. Story 2.3 turned a no-match case into three explicit paths: manual reuse, minimal preset creation, and continue without preset.
- Task-level subtitle template override was scoped correctly. Story 2.4 kept overrides preset-backed, task-local, and visible in task/support context without mutating the underlying preset.
- Review follow-ups were completed rather than left as loose debt. Stories 2.3 and 2.4 both had review findings, both were reverted to active work, fixed, retested, and then approved.

### Quality Signals

- Story 2.1 review outcome: approved with no blocking findings.
- Story 2.2 review outcome: approved with no blocking findings.
- Story 2.3 review outcome: initially changes requested, then approved after follow-ups.
- Story 2.4 review outcome: initially changes requested, then approved after follow-ups.
- Latest story-level validation recorded `pnpm typecheck`, `pnpm test`, and `pnpm build` passing.
- Test count rose from 59 tests in early Epic 2 records to 65 tests by Story 2.4.

## Recurring Patterns

### Positive Patterns to Keep

- Domain ownership stayed coherent: preset logic stayed in `features/presets`, task intake logic stayed in `features/tasks`, and task querying/summary semantics were expanded where consumers needed readable provenance.
- The implementation consistently froze task creation context into task snapshots. This is essential for support, audit, API responses, and future retries.
- Review findings were precise and acceptance-criteria-linked, which made fixes concrete instead of subjective.

### Problems That Repeated

- Confirmation failure recovery was a repeated weak point. Story 2.3 and Story 2.4 both exposed the same class of bug: failed confirm actions could clear the current preview/decision context, forcing the creator back toward the start of the flow.
- Front-end behavior gaps were not always caught by service-level tests first. Both later stories required additional workspace/front-end regression coverage after review.
- Story file tracking missed changed files in both Story 2.3 and Story 2.4, especially shared task intake and event/error files. This is documentation debt, but it also signals that cross-cutting changes are easy to under-record.

## Epic 3 Preview

Epic 3: Recoverable Review and Failure Handling

Planned stories:

- Story 3.1: Low-confidence Review Queue for Creators
- Story 3.2: Failure Explanation and Retry-to-New-Attempt Recovery
- Story 3.3: Support Timeline and Diagnostic Context

Epic 3 builds directly on Epic 1 and Epic 2 state semantics:

- It depends on task lifecycle status being a durable source of truth, especially `awaiting_human_review`, `failed`, and resumed/retried states.
- It depends on task events being rich enough to explain source recognition, preset match/no-match, manual decisions, task-level overrides, review requirements, review resolution, failure, and retry attempts.
- It depends on Epic 2 provenance fields being readable by support and future API-facing layers.

## Required Preparation Before Epic 3

These items should be treated as entry conditions before implementing Epic 3 stories.

### 1. Lock the Human Review Domain Contract

Owner: Architect + Developer

Completion standard:

- Decide whether low-confidence review is represented as a `review_items` table, a task subresource, an action endpoint, or another explicit resource.
- Define stable identifiers for review item, task, segment, decision, actor, and decision status.
- Define how a task transitions into and out of `awaiting_human_review`.
- Define the audit event emitted when review is required and when review is resolved.

Reason:

Architecture already states that human intervention must be modeled as explicit review resources or action endpoints, not vague task status fields. Story 3.1 should not invent this contract ad hoc in UI code.

### 2. Define Failure Reason and Retry Attempt Semantics

Owner: Architect + Developer

Completion standard:

- Define failure phase taxonomy.
- Define machine-readable reason codes and user-readable descriptions.
- Define request/diagnostic identifiers that must be visible to creators and support.
- Define the relationship between original task, failed task state, and new retry attempt.
- Define which source snapshot, preset snapshot, task-level overrides, and review decisions carry into a retry attempt.

Reason:

Story 3.2 requires retry-to-new-attempt recovery. Without an attempt model, retry can accidentally become either a hidden mutation of the failed task or a duplicated task with lost provenance.

### 3. Establish Support Diagnostic Minimum Field Set

Owner: Product Owner + Support/QA + Developer

Completion standard:

- List the fields support must see for preset match/no-match, manual preset reuse/create/continue, subtitle override, review decisions, failure phase, reason code, request ID, and task event timeline.
- Define support access control separately from creator ownership.
- Add a testable authorization rule for support diagnostic access.

Reason:

Story 3.3 is not just a UI timeline. It is a permissioned diagnostic surface that must not weaken task, deliverable, or audit boundaries.

### 4. Add an Error-Recovery Regression Checklist for Epic 3

Owner: QA + Developer

Completion standard:

- Every submit/confirm/retry action in Epic 3 has regression coverage that proves failed submissions preserve the current task, review, failure, or retry context.
- Tests cover both service behavior and user-visible workspace behavior where context retention matters.
- Toast-only error handling is rejected for review, failed task, retry, and support diagnostic flows.

Reason:

Epic 2 repeatedly exposed context loss after failed confirm actions. Epic 3 has more severe consequences if review or failure recovery context disappears.

### 5. Confirm Event Timeline Contract Against Existing `task_events`

Owner: Developer

Completion standard:

- Confirm current task event schema can represent `task.review_required`, `task.review_resolved`, `task.failed`, retry attempt creation, and support timeline entries.
- If current schema is insufficient, introduce the schema change before story-specific UI work.
- Keep top-level task status separate from fine-grained events.

Reason:

Architecture already separates product-level task status from fine-grained `task_events`. Epic 3 should preserve that boundary.

## Risks Entering Epic 3

### High Risk: Context Loss Reappears in Review or Retry Flows

Epic 2 found this issue twice. In Epic 3, the same bug class would be more damaging because the user may lose review decisions, failure explanation context, or retry intent.

Mitigation:

- Make context retention an explicit acceptance criterion in Story 3.1 and Story 3.2.
- Add front-end regression tests for failed review decision submission and failed retry action.

### High Risk: Three Epic 3 Stories Invent Different Event Semantics

Story 3.1, 3.2, and 3.3 all need task events. If each story adds local event payloads independently, support timeline and API readiness will suffer.

Mitigation:

- Define event names and payload shape before implementation.
- Treat event format as a shared cross-module contract.

### High Risk: Retry Mutates Historical Task Context

Epic 2 deliberately preserved task snapshots. Retry must not undermine that by rewriting failed task state or silently changing historical preset/override context.

Mitigation:

- Use a new attempt or equivalent execution instance.
- Preserve original task and attempt lineage.
- Make carry-forward rules explicit.

### Medium Risk: Support View Overexposes Creator Data

Story 3.3 introduces support visibility. Architecture already flags support/ops access as a permission boundary.

Mitigation:

- Add support access tests before broad diagnostic UI.
- Avoid reusing creator-only loaders without explicit authorization checks.

### Medium Risk: Low-Confidence Review Scope Expands Into Editing

Story 3.1 asks for confirmation/handling, not a heavy subtitle editing suite.

Mitigation:

- Keep review decisions focused on the specific low-confidence segment.
- Defer broad editing features unless the epic plan is explicitly changed.

## Does Epic 3 Need Plan Adjustment?

Recommendation: Yes, adjust sequencing and acceptance criteria, but do not change the overall Epic 3 goal.

The current three-story Epic 3 plan is directionally sound. The adjustment needed is a preparation gate before Story 3.1, plus tighter cross-story acceptance criteria.

Recommended adjustment:

- Add an Epic 3 preparation item or mini-story: "Review, Failure, Retry, and Diagnostic Contract Setup".
- Alternatively, make the first task group of Story 3.1 explicitly contract-first and block UI work until the review resource, task status transition, and event payloads are defined.
- Add context retention ACs to Story 3.1 and Story 3.2, mirroring the Epic 2 lesson.
- Add support authorization and diagnostic minimum-field ACs to Story 3.3 before implementation.

This is not a product direction change. It is a sequencing and contract-hardening change needed to avoid predictable integration failures.

## Final Retrospective Conclusion

Epic 2 successfully established the preset-backed launch foundation. The most important learning is that Yakimoji's hard problems are now centered on recoverability, provenance, and explainability. Epic 3 is exactly where those problems become first-class product behavior.

Proceed with Epic 3, but make it contract-first. Do not let low-confidence review, failure recovery, and support diagnostics become three separate local implementations.
