Date: 2026-06-08
Project: Yakimoji
Facilitator: Amelia (Developer)

## Executive Summary

Epic 3 已从故事跟踪层面完成。`sprint-status.yaml` 中 `3-0`、`3-1`、`3-2`、`3-3` 全部为 `done`，`epic-3` 也已标记为 `done`。

这次 Epic 的主要价值不在新增主路径，而在于补齐异常链路的系统韧性与语义一致性。团队先通过 Story 3.0 收敛了 review、failure、retry 与 support diagnostics 的共享 contract，再在 3.1 到 3.3 中分别落地低置信度确认队列、失败解释与新 attempt 重试、以及 support 诊断时间线。

本次 retrospective 的核心发现不是实现质量不足，而是验证可见性不足。Epic 3 已经通过 story-level 测试和 review 建立了较稳的内部 contract，但目前缺少一份面向 Epic 4 的统一 contract test matrix，无法快速回答“Epic 3 的哪些测试在支撑 Epic 4 的 API contract 信心”。

## Recurring Patterns

### Positive Patterns to Keep

- 先定义共享 contract，再进入 story 实现，显著降低了跨故事语义漂移风险。
- 团队对“失败提交后上下文不能丢”这一风险已经形成共同警觉，并转化为实现和测试要求。
- support-only 视角与 creator 视角的分离意识增强，权限边界没有被异常流程稀释。

### Problems and Gaps

- Epic 3 的核心测试证据分散在各个 story 文档中，缺少 Epic 级统一测试矩阵。
- 项目级 `test-summary.md` 过于基础，无法反映 review、failure、retry、diagnostics、auth boundary 等高风险链路覆盖情况。
- 当前产物能证明“故事完成并通过测试”，但不能快速证明“Epic 4 依赖的内部 contract 已系统性验证完成”。

## Epic 4 Preview

Epic 4: API-Integrated Processing Node

Planned stories:

- Story 4.1: API Credential Validation and Task Create Request
- Story 4.2: API Task Status and Result Query Contract
- Story 4.3: Structured Failure and Non-match API Responses

Epic 4 直接依赖 Epic 3 刚刚收紧的内部 contract，尤其依赖以下能力：

- 统一任务状态语义
- 统一失败语义与原因代码
- retry lineage 与新 attempt 语义
- support / creator 权限边界不污染领域模型
- review / failure / retry / diagnostics 可映射为外部 API contract

## Required Preparation Before Epic 4

这些项目被本次 retrospective 明确认定为 Epic 4 的 readiness gate，同时也是正式前置准备任务。未完成前，不建议进入 Epic 4 正式 story 开发。

### 1. 补齐 Epic 3 Contract Test Matrix

Owner: QA + Developer

Completion standard:

- 以 `review / failure / retry / diagnostics / auth boundary / task status semantics` 为分组建立统一测试矩阵
- 每个能力点标出对应测试文件或验证入口，而不是仅写“已测试”
- 额外标注该项覆盖如何支撑 Epic 4 的 `task create`、`status query`、`result query`、`structured failure / non-match response`

Reason:

当前最大缺口不是缺测试，而是缺少面向 Epic 4 的 contract verification visibility。

### 2. 补齐 Internal-to-External API Contract Mapping

Owner: Architect + Developer

Completion standard:

- 整理内部状态、失败分类、retry lineage、non-match 语义到外部 API 字段/response envelope 的映射关系
- 明确哪些内部字段可外露，哪些仅保留为内部诊断语义
- 在 Epic 4 开始前形成可直接用于 story 4.1~4.3 的 contract 草案

Reason:

若没有这层映射，Epic 4 很容易出现“内部语义稳定，但外部 API 语义临时翻译”的问题。

### 3. 补齐 API Credentials 认证边界测试策略

Owner: Architect + QA + Developer

Completion standard:

- 明确 `api_credentials` 与 Web session/cookie 的隔离边界
- 明确 401/403 行为、无凭证/无效凭证/过期凭证的测试策略
- 明确不存在越权读任务、泄漏受保护业务数据的验证项

Reason:

Epic 4 的风险不仅在返回结构，也在认证模型混用。一旦边界不清，后续 contract 会失真。

### Interpretation

## Key Takeaways

1. Epic 3 的主要价值是把异常链路从“能跑”提升到“有统一 contract、可解释、可恢复”。
2. 团队已经吸收 Epic 2 retrospective 的关键系统性教训，并在 Epic 3 中转化为 contract-first 与回归护栏。
3. 当前最大缺口是 Epic 级验证可见性，而不是明显的功能或测试缺失。
4. Epic 4 开工前必须补齐 contract test matrix、internal-to-external API mapping、以及 API credential auth test strategy。
